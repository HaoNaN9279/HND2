# 命运2 离线数据库查询规范

## 1. 数据库生命周期管理

### 1.1 前置检查
在执行任何数据库查询任务前，**必须**先验证数据库状态：

```bash
python scripts/download_db.py --check
```

- 如果返回码为 0（数据库已是最新）→ 继续查询
- 如果返回码非 0（数据库不存在或需要更新）→ 先运行 `python scripts/download_db.py` 下载/更新，再执行查询

### 1.2 首次设置
1. 前往 https://www.bungie.net/en/Application 注册应用获取 API Key
2. 设置环境变量：`set BUNGIE_API_KEY=你的密钥`（Windows）
3. 运行 `python scripts/download_db.py` 下载数据库（约 340MB）

### 1.3 强制更新
需要最新数据时：`python scripts/download_db.py --force`

## 2. 查询工作流

查询任务必须经过四个阶段：**需求分析 → 策略制定 → 查询执行 → 结果校验**。每个阶段有明确产出物，缺一不可。

---

### 2.1 需求分析（强制拆解）

接收到查询需求后，按以下步骤拆解。**禁止跳过、禁止猜测、禁止凭经验假设**。

**步骤一：拆解为原子问题**

将用户需求拆解为不可再分的最小查询单元。每个单元只查一个信息点。

> **正确示范**：
> - "查找手炮类武器中的以下Perk列表" → 拆解为：①哪些手炮 ②每把手的Perk池 ③指定Perk是否存在
> - "有哪些金装防具" → 拆解为：①异域品质筛选 ②防具类型筛选 ③获取列表

**步骤二：识别缺失信息**

对于每个原子问题，列出回答它所需的全部参数。缺失的参数必须形成问题询问用户。

> **必须询问的情况**：
> - 关键词不明确（"查找带增伤的武器" → 什么叫"增伤"？具体Perk名是什么？）
> - 范围不明确（"查一些防具" → 哪些？所有防具还是某个类型？）
> - 过滤条件不明确（"最近赛季的武器" → 哪个赛季？按什么标准算"最近"？）
> - 排序/数量要求不明确（"列出几把武器" → 几把？按什么排序？）

**步骤三：列出查询参数清单**

输出一份参数检查单：

```markdown
## 查询参数清单
- [ ] 目标表/定义类型：___（如不明确 → 询问）
- [ ] 筛选条件：___（如不明确 → 询问）
- [ ] 排序方式：___（如不明确 → 询问）
- [ ] 结果数量限制：___（如不明确 → 询问）
- [ ] 输出字段：___（如不明确 → 询问）
```

参数清单有任何一项不明确，**必须**向用户提问。不得替用户做决定。

---

### 2.2 查询策略制定

**前置条件**：参数清单全部确定后，方可进入策略制定。

**步骤一：查阅 Schema**

打开 `database_schema.md`，根据目标查询内容定位对应的表。校验要点：

1. **表定位**：根据字段名和表描述确定哪个表包含所需数据
2. **字段验证**：确认所需字段确实存在于目标表的 JSON 中（看字段列表）
3. **行数评估**：查看目标表的行数，评估数据规模
4. **Hash 引用链路**：字段名含 `Hash` 后缀的表示引用其他表，需规划多表查询链路

**步骤二：制定查询策略**

根据表的数据量选择策略：

| 数据量 | 策略 | 说明 |
|--------|------|------|
| **≥10,000 行**（5张表：Unlock/InventoryItem/Icon/Collectible/Objective） | **必须加WHERE/LIKE过滤**，缩小扫描范围。禁止全表扫描。先用 `COUNT(*)` 确认数据量，再基于 `id` 或 `json LIKE` 做筛选，必要时分页处理 |
| **1,000–9,999 行**（11张表） | 建议加过滤条件，可用 `json LIKE %keyword%` 搜索，但尽量用精准字段 |
| **100–999 行**（17张表） | 可全表扫描后过滤 JSON，关注 `index` 字段排序 |
| **10–99 行**（29张表） | 全表扫描无压力，可一次性拉取 |
| **1–9 行**（19张表） | 极小表，直接全量拉取 |
| **0 行**（2张表：ReportReasonCategory/RewardSource） | 空表，跳过查询 |

**步骤三：规划 Hash 引用解析**

当涉及跨表引用（如一个表存 hash，另一个表存名称），需在策略中明确：

- 哪些 hash 需要解析
- 哪张表负责解析（目标定义表）
- 预计涉及的记录数（避免 N+1 查询，尽量用 `IN` 或批量查询）

**步骤四：输出查询策略**

```markdown
## 查询策略
- 目标表：___（行数：___） → 策略：___
- Hash 解析链路：___
- 预计查询次数：___
- 风险点：___
```

---

### 2.3 查询执行

- **默认查询当前版本数据**：所有查询默认针对用户提问时所在版本的赛季内容。查询前应通过 §1.1 确保数据库为最新状态。若用户明确提到历史赛季/版本（如"S15时期的Perk"），需判断当前 Manifest 是否仍包含该数据（Bungie 每个赛季刷新 Manifest，不保留历史版本）。
- 任何内容都从数据表中获取，**绝对禁止**凭空捏造信息。
- 查询数据时确保查找所有需要查找的数据，不允许跳过，漏查。
- 优先使用 `scripts/db.py` 中的通用工具函数。
- 复杂查询（多表、跨 JSON 字段）使用临时脚本，放在 `scripts/temp/` 下，完成后清理。
- 使用 `uv run python` 执行，优先使用标准库 `sqlite3` + `json`。
- 逐项输出查询结果，并以表格形式汇总（Markdown 表格）。

---

### 2.4 结果反向校验

查询完成后，**必须**对结果做回归校验。不允许宣称"查询完成"而未验证。

**校验清单**：

```markdown
## 结果校验
- [ ] **不缺**：将结果数量回查数据库验证。例如查询"所有手炮"，先查 `SELECT COUNT(*) FROM DestinyInventoryItemDefinition WHERE json LIKE '%手炮%'`，对比结果数量与输出数量是否一致
- [ ] **不错**：随机抽取 2–3 条结果，从数据库直接调取 JSON 原始数据，逐字段对比输出值与库中值是否一致
- [ ] **不多**：确认没有引入不属于查询范围的记录（如查"手炮"时没有混入"脉冲步枪"）
- [ ] **Hash 引用正确**：如果解析了 hash → 名称，随机抽一个 hash 去对应定义表验证名称是否正确
```

**发现不一致的处理流程**：
1. 定位问题源（是筛选条件遗漏？还是 hash 解析错误？）
2. 修正查询脚本
3. 重新执行并再次校验
4. 直到校验清单全绿通过

**校验通过后方可交付结果**。

## 3. 查询结果输出规范

### 3.1 格式要求
所有数据库查询结果**必须**输出为 Markdown 文件。MD 文件应当：
- UTF-8 编码，无 BOM
- 包含完整的查询说明和结果表格
- 文件命名：按查询内容英文命名（如 `hand_cannons_with_incandescent.md`），存于项目根目录或 `results/` 目录
- 禁止使用 HTML 格式输出（除极少数用户明确要求外）

### 3.2 MD 模板要求
```markdown
# 查询标题

> **查询时间**：YYYY-MM-DD HH:MM
> **查询条件**：[此处写用户原始需求摘要]
> **来源**：destiny2_zh-chs.sqlite3
> **结果行数**：N

## 查询参数
- [参数1]: 值1
- [参数2]: 值2
...

## 查询结果

| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 值1 | 值2 | 值3 |
| ... | ... | ... |

## 校验记录
- 数据量验证：✓（N 条 vs 数据库 N 条）
- 抽样验证：✓（抽检 n 条全部通过）
- 额外记录验证：✓（无多余记录）
```

### 3.3 查询代码规范
- 通用查询脚本放在 `scripts/` 目录下。
- 临时查询脚本统一放在 `scripts/temp/` 目录下，查询完成后**必须**清理删除。
- 每次执行完查询任务，将查询任务中使用的查询方法提取出来，增加/覆盖/改进现有的通用查询脚本。
- 写临时查询脚本时，优先使用通用查询脚本。
- 查询脚本使用 `uv run python` 执行
- 优先使用 Python 标准库 `sqlite3` + `json`，不引入额外依赖

### 3.4 通用查询工具
现有通用工具位于 `scripts/db.py`，提供：

| 函数 | 功能 |
|---|---|
| `get_conn()` | 获取数据库连接 |
| `load_json(val)` | 解析 BLOB/TEXT 为 JSON |
| `search_perk(cur, keyword)` | 按关键词搜索 Perk（名称+描述） |
| `search_items(cur, keyword)` | 按关键词搜索物品 |
| `resolve_perk_name(cur, plug_hash)` | 从 plug hash 解析 Perk 名称 |
| `resolve_perk_with_description(cur, plug_hash)` | 解析 Perk 名称+效果描述 |
| `find_weapons_with_perks(cur, plug_hashes)` | 查找含指定 Perk 组合的武器 |
| `get_weapon_socket_analysis(cur, weapon_id)` | 分析武器所有插槽和 Perk 池 |
| `SLOT_TYPES` 字典 | 武器槽类型 hash → 名称映射 |
| `to_signed(hash)` / `to_unsigned(id)` | 有符号/无符号 ID 互转 |

## 4. 数据库编码注意事项

- `id` 列存储的是有符号 32 位整数。JSON 中的 hash 为无符号表示，查询时需转换
- ID 转换方法：Python 中 `hash_value if hash_value < 2**31 else hash_value - 2**32`
- 或直接使用带符号整数字面量（如 `-1902153673`）
